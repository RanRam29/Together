# WP9 — ‏D31: ממשק הורה שני + D44: הגנת צילום מסך

> בעלות: Cursor (ליבה), Antigravity (סעיף 3 — RPC אירוע צילום). תלות: שלב 1 סגור. D31 הוא **חוסם השקה** — קודם ל-D44.

## חלק א' — D31: ממשק הורה שני (הצד השרתי קיים ופרוס)

RPC-ים קיימים (מ-`20260712210000_secondary_parent.sql`, כולם מוקשחים): `invite_secondary_parent(p_child_id, p_phone)` · `accept_parent_invitation(p_invitation_id)` · `update_secondary_permissions(p_child_id, p_permissions jsonb)` · `remove_secondary_parent(p_child_id)` · `transfer_primary_parent(p_child_id)`. קריאת הזמנות ממתינות: טבלת `parent_invitations` (RLS לפי טלפון מאומת של המוזמן — `20260713000000` סעיף M2).

### מסכים/זרימות (מודל ההרשאות מ-D31 — הורה ראשי פועל לבד; משני מקבל התראה אחרי-מעשה על פעולות קריטיות):
1. **הזמנה (צד מזמין):** בפרופיל הילד — סעיף "הורה נוסף": מצב ריק עם כפתור "הזמנת הורה נוסף" → טופס טלפון (ולידציית 972…) → `invite_secondary_parent` → מצב "ממתין לאישור" עם אפשרות ביטול (`remove_secondary_parent`).
2. **קבלה (צד מוזמן):** אחרי התחברות OTP — אם קיימת שורה ב-`parent_invitations` בסטטוס ממתין לטלפון שלו: כרטיס הזמנה במסך הבית ("X הזמין/ה אותך להיות שותף/ה בליווי של Y") → `accept_parent_invitation`. חשוב: המוזמן חייב להיות במסלול תפקיד הורה; אם נרשם כמשלבת — להציג הסבר, לא לקרוא ל-RPC.
3. **ניהול (צד ראשי):** מסך הרשאות להורה המשני — מתגים לפי מפתחות `secondary_parent_permissions` הקיימים בסכמה + המפתח `manage_visibility` (נצרך ע"י WP8). שמירה דרך `update_secondary_permissions` בלבד (עדכון ישיר ייחסם ע"י הטריגר). הסרה + העברת בעלות (`transfer_primary_parent`) מאחורי אישור כפול.
4. **חוויית ההורה המשני:** רואה את הילד לפי הרשאותיו; פעולות שאין לו — מוסתרות (לא מנוטרלות-אפורות). ההתראות אחרי-מעשה על פעולות קריטיות (סיום match, מחיקת פרופיל, שינוי הרשאות) כבר נשלחות מהשרת — לוודא שהאפליקציה מציגה אותן.

### בדיקות (ידני + בדיקת רכיבים):
שני מכשירים/פרופילים: הזמנה → קבלה → צפייה לפי הרשאה → צמצום הרשאה ואימות שהמסך משתקף → הסרה. ולידציה: אי-אפשר להזמין את עצמך, טלפון קיים כהורה משני לילד אחר מותר.

## חלק ב' — D44: הגנת צילום מסך בתיק הילד

היקף: כל מסך שמציג נתוני `child_details` (תיק ילד TIER2/3, כולל מסך הפרטים למשלבת).

1. **אנדרואיד — חסימה אמיתית:** `expo-screen-capture` → `usePreventScreenCapture()` במסכים שבהיקף (FLAG_SECURE — צילום/הקלטה מחזירים שחור).
2. **iOS — זיהוי + תיעוד:** `ScreenCapture.addScreenshotListener` במסכים שבהיקף → קריאה ל-RPC חדש (סעיף 3) + הודעה עדינה למשתמש לפי `06-COPY-TONE` (לא מאשימה: תזכורת שהמידע רגיש ומוגן).
3. **Antigravity — RPC קטן:** `log_screenshot_event(p_child_id UUID)` — SECURITY DEFINER + `search_path`; מאמת שלקורא יש גישת TIER≥2 לילד (או הורה); כותב ל-`audit_log` ‏(resource=`child_details`, action=`screenshot_detected`, tier מהפונקציה הקיימת). אירועים אלה יופיעו בהמשך בדוח ה-audit של האדמין (WP10).
4. לא לחסום את כל האפליקציה — רק את המסכים שבהיקף (מסכי browse/TIER0 חופשיים).

## הגדרת גמר
D31: זרימה מלאה עברה על שני מכשירים אמיתיים · `tsc` נקי. D44: אימות ידני אנדרואיד (צילום שחור) + iOS (שורת audit נכתבה) · pgTAP קצר ל-RPC (משתמש זר נדחה).
